## Overview

Equipment presets are named collections of equipment slugs. Two types:

1. **System presets** (`is_system = true`) — seeded at deploy time, read-only via `GET /equipment-presets/system`. Examples: "Тренажёрный зал" (full gym), "Домашняя тренировка" (home), "Уличная площадка" (outdoor/street), "Только вес тела" (bodyweight only).
2. **User presets** (`is_system = false`) — created/edited/deleted by users via CRUD. Can be created from scratch or cloned from a system/user preset.

Presets are used:
- In `/workout-dialog` step `equipment` — user can select a preset instead of picking individual items
- In `/workout-milp/generate` and `/workout-milp/weekly-plan` — `availableEquipment` can come from a preset slug
- In user profile — `user.metadata.defaultEquipmentPresetId` stores the user's preferred default

## Data Model

### New table: `equipment_presets`

```sql
CREATE TABLE equipment_presets (
  id TEXT PRIMARY KEY,
  user_id TEXT REFERENCES users(id) ON DELETE CASCADE,  -- NULL for system presets
  name TEXT NOT NULL,
  slug TEXT NOT NULL,
  is_system BOOLEAN NOT NULL DEFAULT false,
  equipment_slugs TEXT[] NOT NULL DEFAULT '{}',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(user_id, slug)
);
```

- System presets have `user_id = NULL`
- User presets have `user_id` set
- `equipment_slugs` is an array of equipment slugs from the `equipments` table
- The UNIQUE constraint on `(user_id, slug)` means:
  - Multiple users can have presets with the same slug
  - System presets (user_id=NULL) are unique by slug globally

### Entity

```typescript
// src/entities/equipment-preset.entity.ts
export interface EquipmentPreset {
  id: string;
  userId?: string;
  name: string;
  slug: string;
  isSystem: boolean;
  equipmentSlugs: string[];
  createdAt: string;
  updatedAt: string;
}
```

## Seeded System Presets

Seeded via a migration SQL file. The `equipment_slugs` arrays reference real slugs from the `equipments` table. At implementation time, read the actual slugs from the DB to populate these arrays.

Known slugs from test data: `barbell`, `dumbbell`, `cable`. The DB has many more (machines, benches, bands, etc.).

Preset definitions (names are final, slugs TBD from DB):

| Preset | slug | Description | Expected equipment categories |
|--------|------|-------------|------------------------------|
| Тренажёрный зал (полный) | `gym-full` | Всё что есть в стандартном зале | barbells, dumbbells, cables, machines, kettlebells, benches, pull-up bars, cardio |
| Тренажёрный зал (базовый) | `gym-basic` | Минимальный набор зала | barbell, dumbbells, bench, pull-up bar, cable |
| Домашняя тренировка | `home` | Типичный домашний набор | dumbbells, resistance bands, pull-up bar, kettlebell, stability ball |
| Только вес тела | `bodyweight-only` | Без оборудования | `[]` (empty array) |
| Уличная площадка | `outdoor` | Воркаут-площадка | pull-up bar, dip station, resistance bands |

**Implementation note**: Write a separate seed script `sql/seed-equipment-presets.ts` that:
1. Reads all existing equipment slugs from the `equipments` table
2. Matches them to the preset definitions by slug pattern or name
3. Inserts the presets with valid slugs only (skips slugs that don't exist)
4. Uses `ON CONFLICT DO NOTHING` for idempotency

## API Endpoints

### System Presets (read-only)

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| GET | `/equipment-presets/system` | No | All system presets |

**Response**:
```json
[
  {
    "id": "preset-gym-full",
    "name": "Тренажёрный зал (полный)",
    "slug": "gym-full",
    "isSystem": true,
    "equipmentSlugs": ["barbell", "dumbbell", "cable", ...],
    "equipmentDetails": [
      { "slug": "barbell", "name": "Barbell" },
      { "slug": "dumbbell", "name": "Dumbbell" }
    ]
  }
]
```

`equipmentDetails` is an optional join that returns the human-readable equipment names alongside slugs. Included when `?includeDetails=true` query param is set.

### User Presets (CRUD)

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| GET | `/equipment-presets` | JWT | All user presets |
| POST | `/equipment-presets` | JWT | Create preset |
| PATCH | `/equipment-presets/:id` | JWT | Update preset |
| DELETE | `/equipment-presets/:id` | JWT | Delete preset |
| POST | `/equipment-presets/clone/:sourceId` | JWT | Clone preset (system or user) |

**Create Body**:
```json
{
  "name": "Мой зал",
  "slug": "my-gym",
  "equipmentSlugs": ["barbell", "dumbbell", "cable", "bench"]
}
```

**Clone Body**:
```json
{
  "name": "Мой зал (кастомизированный)",
  "slug": "my-gym-custom"
}
```
Cloning copies `equipmentSlugs` from the source (system or user preset). The clone always becomes a user preset (`isSystem = false`).

**Update Body** (all fields optional):
```json
{
  "name": "Обновлённый зал",
  "equipmentSlugs": ["barbell", "dumbbell", "cable", "bench", "kettlebell"]
}
```

## Dialog Integration

### Current behavior
The `equipment` step in `/workout-dialog` shows all individual equipment items from the DB as a multi-choice list.

### New behavior
Add a **new step** `equipment_preset` before the `equipment` step in the dialog flow. This gives the user two paths:

**Path A: Select a preset**
1. User sees system + user presets as single_choice options
2. Selecting a preset fills `availableEquipment` directly
3. The `equipment` step is **skipped**

**Path B: Pick individually**
1. User selects "Выбрать вручную" option
2. Falls through to the existing `equipment` multi_choice step

### Step addition

Updated step order:
```
plan_type → goal → experience → focus_muscles → equipment_preset → equipment → frequency → days → duration
```

`equipment_preset` step definition:
```
question: "Выберите оборудование или пресет"
inputType: single_choice
options:
  - System presets (gym-full, gym-basic, home, bodyweight-only, outdoor)
  - User's presets (dynamic from DB)
  - "Выбрать вручную" → manual mode (skip to equipment step)
skip conditions:
  - If user has metadata.defaultEquipmentPresetId → auto-fill and skip
```

### Implementation
- `StepId` type: add `'equipment_preset'`
- `STEP_ORDER`: insert `equipment_preset` before `equipment`
- `STEP_DEFINITIONS`: add definition with dynamic options from DB
- `applyAnswer`: when user selects a preset slug → `collected.availableEquipment = preset.equipmentSlugs`
- `getNextStep`: if `equipment_preset` fills `availableEquipment` → skip `equipment` step
- The `equipment` step becomes the fallback for manual selection

## MILP Integration

### `/workout-milp/generate` and `/workout-milp/weekly-plan`

Add optional `equipmentPresetId` field to both DTOs. When provided:
1. Load preset from DB
2. Use `preset.equipmentSlugs` as `availableEquipment`
3. If `availableEquipment` is also explicitly provided → preset takes precedence (or merge? → **preset takes precedence**)

This is a convenience shortcut so the client can send `equipmentPresetId` instead of the full array.

### DTO changes

`GenerateWorkoutDto`:
```typescript
@ApiPropertyOptional({ example: 'preset-gym-full' })
@IsOptional()
@IsString()
equipmentPresetId?: string;
```

`GenerateWeeklyPlanDto`:
```typescript
@ApiPropertyOptional({ example: 'preset-gym-full' })
@IsOptional()
@IsString()
equipmentPresetId?: string;
```

### Controller logic
```typescript
// In generate():
let availableEquipment = dto.availableEquipment;
if (dto.equipmentPresetId) {
  const preset = await presetsRepository.findById(dto.equipmentPresetId);
  if (preset && (preset.isSystem || preset.userId === user.id)) {
    availableEquipment = preset.equipmentSlugs;
  }
}
availableEquipment ??= fullUser?.metadata?.availableEquipment ?? [];
```
