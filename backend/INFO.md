# Аналитический обзор архитектуры Fitness App

## 1. Введение

Рассматриваемый backend реализует систему генерации тренировочных сценариев двух уровней сложности:

1. генерация одной тренировки;
2. генерация недельного плана тренировок.

Функционально система состоит из трёх взаимосвязанных контуров:

1. HTTP API, через который пользователь получает доступ к генерации, метрикам и диалогу;
2. диалоговый слой, собирающий параметры для последующей оптимизации;
3. оптимизационный слой MILP, в котором формализован выбор упражнений, подбор сплита и распределение нагрузки.

Ключевая особенность решения состоит в том, что оно не ограничивается статическим шаблонизатором тренировок. Вместо этого используется многоуровневая логика отбора, где решения принимаются на основе данных о метаданных упражнений, истории использования, усталости мышечных групп, цели пользователя, уровня подготовки, пола и доступного оборудования.

## 2. Архитектура API и роль маршрутов

### 2.1 Общая структура

API организован как набор доменных модулей. Для рассматриваемой подсистемы особенно важны:

1. `workout-milp` - оптимизационная генерация тренировок;
2. `workout-dialog` - последовательный сбор параметров для генерации;
3. репозитории, обеспечивающие доступ к упражнениям, пользователям, блокам и сессиям.

Все основные маршруты `workout-milp` и `workout-dialog` защищены JWT-авторизацией, что позволяет учитывать индивидуальные данные пользователя при расчёте планов.

### 2.2 Эндпоинты workout-milp

Подсистема `workout-milp` предоставляет три основных операции:

1. `POST /workout-milp/generate` - генерация одной тренировки;
2. `POST /workout-milp/metrics` - вычисление метрик для заданного набора упражнений;
3. `POST /workout-milp/weekly-plan` - построение недельного плана.

С архитектурной точки зрения это важно: генератор одной тренировки является базовой атомарной операцией, weekly-plan строится как надстройка над ним, а metrics служит аналитическим инструментом для оценивания результата.

### 2.3 Эндпоинты workout-dialog

Подсистема `workout-dialog` реализует пошаговый диалог:

1. `POST /workout-dialog/start` - создание нового диалога;
2. `POST /workout-dialog/:dialogId/answer` - отправка ответа и переход к следующему шагу;
3. `GET /workout-dialog/:dialogId` - получение текущего состояния;
4. `DELETE /workout-dialog/:dialogId` - удаление диалога.

Диалог не генерирует тренировку напрямую. Его задача - собрать параметры в нормализованном виде, после чего клиент может вызвать `generate` или `weekly-plan`.

## 3. Диалоговый слой как механизм параметризации MILP

### 3.1 Назначение диалога

Диалоговая система выполняет функцию семантического предсборщика входных данных. Она уменьшает вероятность неконсистентного запроса, так как пользователь последовательно отвечает на вопросы, а сервер хранит промежуточное состояние.

### 3.2 Последовательность шагов

Логика `WorkoutDialogService` задаёт фиксированную последовательность шагов:

1. выбор типа плана: одна тренировка или неделя;
2. цель (`goal`);
3. уровень опыта (`experience`);
4. акцент на мышечные группы (`focus_muscles`) - только для single workout;
5. выбор пресета оборудования или ручной настройкой;
6. выбор оборудования вручную;
7. частота тренировок в неделю (`frequency`) - только для weekly;
8. выбор дней недели (`days`) - только для weekly;
9. длительность тренировки (`duration`).

Таким образом, диалоговый слой представляет собой управляющую оболочку над MILP-моделью, а не альтернативу ей.

### 3.3 Скрытая логика заполнения параметров

Диалог предварительно заполняет ответы из профиля пользователя:

1. `goal`;
2. `experienceLevel`;
3. `availableEquipment`.

Это означает, что система использует накопленное пользовательское состояние как априорное знание. С математической точки зрения такой подход уменьшает размер пространства поиска для MILP за счёт сокращения числа неопределённых переменных на входе.

### 3.4 Представление ответов

Для мультивыборных шагов используется строка с разделением по запятой. Повторная отправка значения реализует toggle-механику. Это важно для UX, но также влияет на консистентность данных: итоговое множество мышц, оборудования или дней формируется как множество без дублей.

## 4. Концептуальная схема MILP-ядра

Система реализует две связанные оптимизационные модели:

1. Workout MILP - выбор упражнений и параметров повторения/подходов для одной тренировки;
2. Weekly Process MILP - распределение нескольких тренировок по неделе с учётом восстановления и кумулятивной нагрузки.

Хотя в коде используется гибридная схема «MILP + эвристики», математическая постановка в документации и фактическая реализация имеют одну общую структуру:

1. ранжирование кандидатов по весу;
2. введение жёстких ограничений;
3. обработка неразрешимости через жадный fallback;
4. накопление и перенос состояния между итерациями.

## 5. Workout MILP: формализация и закономерности

### 5.1 Смысл задачи

Цель модели - выбрать набор упражнений `E*` и сопутствующие параметры так, чтобы:

1. тренировка укладывалась во временной лимит;
2. покрывала целевые мышечные группы;
3. учитывала противопоказания и доступное оборудование;
4. была согласована с целью, опытом, полом и тренировочной историей;
5. обеспечивала разнообразие и не усиливала усталость сверх порога.

### 5.2 Переменные решения

Из документации и реализации следует следующий набор логических переменных:

1. `x_e ∈ {0,1}` - выбран ли элементарный кандидат `e`;
2. `s_e` - число подходов у упражнения `e`;
3. `z_{t,e}` - принадлежность упражнения тренировке `t` в недельном контуре.

Для одной тренировки ключевой дискретной переменной является `x_e`, а `s_e` задаётся не как произвольная переменная, а как производная величина, зависящая от класса упражнения и цели.

### 5.3 Целевая функция

В документе MILP цель записана как:

```text
max Z = Σ x_e * w_e - Σ x_e * riskPenalty(e) - Σ x_e * contraPenalty(e)
```

В коде эта идея реализована через вычисление веса `w_e` и последующую подачу его в solver как коэффициента objective.

Итоговый вес строится по каскадной схеме:

```text
w_e = w_data(e) * (1 + δ * V_e) * (1 - ε * P_e)
```

где:

1. `w_data(e)` - базовая информативная компонента;
2. `V_e` - коэффициент разнообразия;
3. `P_e` - штраф за усталость.

### 5.4 Data-driven часть

Документальная формула:

```text
w_data(e) = (1 + α1 * f_complexity(e)) * (1 + α2 * f_frequency(e)) * (1 + α3 * f_affinity(e))
```

В коде это соответствует следующей интерпретации:

1. `f_complexity(e)` - близость сложности упражнения к текущей фазе;
2. `f_frequency(e)` - эвристическая константа частотной компоненты;
3. `f_affinity(e)` - совпадение с фазой периодизации.

Следовательно, базовый вес является мультипликативной моделью предпочтений. Она обладает важным свойством: если хотя бы один из факторов сильно снижен, итоговый вес уменьшается не линейно, а композиционно. Это делает модель чувствительной к неблагоприятным признакам.

### 5.5 Разнообразие

Разнообразие задаётся законом:

```text
V_e = min(1, sessionsSinceLastUse(e) / diversity_window)
```

В коде используется эквивалентная эвристика через `usedExercises` и окно `DIVERSITY_WINDOW = 4`.

Интерпретация закономерности проста: чем реже упражнение использовалось в недавнем прошлом, тем выше его шанс попасть в новую тренировку. Это предотвращает монотонность плана и повышает вариативность стимуляции.

### 5.6 Штраф за усталость

В документе:

```text
P_e = min(1, Σ I(e,m) * max(0, F_m - θ))
```

В реализации это сводится к суммированию усталости по мышцам, нормализованной через вес мышцы в упражнении. После этого коэффициент `ε` снижает итоговую привлекательность упражнения.

Функциональный смысл такой:

1. пока усталость мышцы ниже порога `θ`, штраф отсутствует;
2. как только мышцы выходят за порог, упражнения, нагружающие их, дискриминируются;
3. штраф не бинарный, а градуальный, что улучшает устойчивость модели.

### 5.7 Риск и противопоказания

В теоретической постановке:

```text
riskPenalty(e) = riskLevel(e)
```

```text
contraPenalty(e) = Σ (1 - k(severity(c)))
```

Практически это означает:

1. `forbidden` противопоказания исключают упражнение из множества кандидатов;
2. `not_recommended` создаёт штраф с коэффициентом 0.5;
3. `low_weight` создаёт мягкий штраф с коэффициентом 0.8.

Таким образом, система сочетает фильтрацию и ранжирование. Это классическая двухступенчатая схема: сначала отсекаются недопустимые решения, затем упорядочиваются допустимые.

## 6. Workout MILP: система ограничений

### 6.1 Ограничение на число упражнений

```text
Σ x_e = N
```

Это центральное ограничение модели. Оно фиксирует размер решения и делает задачу не просто задачей отбора, а задачей отбора точно заданной мощности.

### 6.2 Ограничение по времени

```text
Σ x_e * timeCostSec(e) * s_e <= sessionDurationMin * 60
```

Фактически это ограничение превращает длительность в ресурс. Каждый кандидат «потребляет» часть временного бюджета. В коде время дополнительно уточняется через `DEFAULT_PER_SET_TIME_SEC` и `restBetweenSetsSec`.

### 6.3 Ограничение по обязательным мышцам

```text
Σ x_e * I(e,m) >= 1,  m ∈ mandatoryMuscles
```

Это ограничение обеспечивает структурную полноту тренировки. Важно, что `mandatoryMuscles` формируется не только из пользовательских фокусов, но и из расширения мышечных групп через иерархию.

### 6.4 Ограничение по оборудованию

Упражнение допускается только если его оборудование пересекается с доступным набором либо если упражнение относится к bodyweight/stretch типу. В коде это реализовано как жёсткая фильтрация кандидатов до запуска solver.

### 6.5 Ограничение по усталости

```text
Σ x_e * I(e,m) <= fatigueLimit(m)
```

В реализации предел установлен как `FATIGUE_LIMIT = 3.0`. Это означает, что система рассматривает усталость как локальный лимит, применяемый к каждой мышечной группе отдельно.

### 6.6 Ограничение на вариативность

```text
Σ_{variationGroup(e)=g} x_e <= 1
```

Это предотвращает выбор упражнений-аналоги внутри одной группы заменяемости. Иначе оптимизация могла бы формально удовлетворить целям, но фактически выдать слишком похожие движения.

### 6.7 Баланс push/pull

Документ задаёт баланс через две неравенства:

```text
Σ push - Σ pull <= 1
Σ pull - Σ push <= 1
```

В коде это выражено через специальные коэффициенты `pushPullBalance` и `pullPushBalance`. Смысл состоит в том, что программа удерживает механическую симметрию нагрузки и не допускает существенного перекоса в одну сторону.

## 7. Недельный контур Weekly Process MILP

### 7.1 Функциональная роль

Недельный процесс не дублирует одиночную генерацию, а компонуется над ней. Он решает задачу более высокого уровня:

1. выбрать стратегию сплита;
2. распределить типы сессий по дням;
3. передать в Workout MILP уточнённые входы для каждой тренировки;
4. учесть межсессионную усталость и недельный объём.

### 7.2 Целевая функция

В теоретической записи:

```text
max Z_week = Σ sessionScore(t) - overloadPenalty - imbalancePenalty
```

Хотя в коде не строится единая глобальная MILP-модель на всю неделю, семантика модели сохраняется: каждую сессию можно интерпретировать как частный вклад в недельный функционал качества.

### 7.3 Матрица сплитов

Ключевая структура weekly-модуля - `SPLIT_STRATEGIES`. Она отображает пару:

```text
(trainingCountPerWeek, experienceLevel) -> последовательность session slots
```

Порядок сессий важен, поскольку он влияет на:

1. усталость, которая переносится между днями;
2. подачу целевых мышечных групп;
3. распределение нагрузки по неделе.

Примеры стратегий:

1. `2-beginner -> full_body, full_body`;
2. `3-intermediate -> push, pull, legs`;
3. `4-advanced -> push, pull, legs, full_body`;
4. `6-* -> push, pull, legs, push, pull, legs`.

### 7.4 Переход от сплита к мышечному таргетингу

Каждый session slot имеет собственную карту мышечного фокуса:

1. `full_body` - грудь, спина, ноги, плечи, руки, кор;
2. `upper` - грудь, спина, плечи, руки;
3. `lower` - квадрицепсы, бицепсы бедра, ягодицы, икры, кор;
4. `push` - грудь, плечи, трицепс;
5. `pull` - спина, широчайшие, верх спины, трапеции, бицепс;
6. `legs` - ноги, ягодицы, икры, приводящие.

Это показывает, что weekly-процесс не является произвольным перечислением дней, а представляет собой дискретную схему периодизации микроплана.

### 7.5 Динамика усталости между сессиями

Документ задаёт рекуррентное соотношение:

```text
F_m(t+1) = F_m(t) * exp(-λ * restDays) + Σ I(e,m) * s_e
```

где `λ = 0.345`.

Это одно из наиболее важных уравнений в системе, поскольку оно придаёт модели временную память. Следствия:

1. усталость экспоненциально убывает с отдыхом;
2. более длинный перерыв сильнее снижает остаточную усталость;
3. новые нагрузки накладываются поверх остаточного состояния.

Именно поэтому weekly-генерация является не статической, а рекурсивной моделью.

### 7.6 Распределение дней

Алгоритм `assignDays` использует упорядочивание доступных дней по хронологии недели и затем выбирает либо:

1. равномерное распределение, если дней меньше, чем сессий;
2. тот же набор, если количество совпадает;
3. наиболее разнесённую подвыборку, если дней больше.

Это означает, что календарная дискретизация решается не MILP-формулой, а эвристическим оптимизатором подзадачи размещения.

### 7.7 Специфика женской и мужской логики

В weekly-контуре учитывается пол пользователя:

1. для female в нижних и full-body слотах дополнительно усиливаются `glutes` и `hamstrings`;
2. для male усиливаются верхние мышцы: `chest`, `shoulders`, `lats`.

На уровне модели это модифицирует множество обязательных мышц и весовые коэффициенты выбора упражнений. На уровне биомеханической интерпретации это можно рассматривать как мягкое априорное предпочтение по распределению объёма.

## 8. Накопительная модель объёма и утомления

### 8.1 Недельный объём

Объём по мышцам вычисляется как суммарное число подходов, накопленное за последнюю неделю.

В коде недельный объём используется для нормализации приоритета упражнений:

1. если мышца уже достигла scaled max, упражнения на неё штрафуются;
2. если мышца значительно недополучила объём, она получает бонус.

Формально это соответствует правилу:

```text
если weeklyDone >= scaledMax -> w_e *= 0.3
если weeklyDone < 0.5 * scaledMin -> w_e *= 1.3
```

Это добавляет системе адаптивность и предотвращает локальную перегрузку отдельных мышц.

### 8.2 Фатиг-модель как накопитель состояния

Фатиг-модель не является простой штрафной переменной. Она играет роль состояния, которое:

1. переносится от тренировки к тренировке;
2. уменьшается после отдыха;
3. пополняется новой нагрузкой.

Это уже ближе к динамической системе управления, чем к одношаговой оптимизации.

## 9. Жадный fallback как гарантия работоспособности

### 9.1 Причина fallback

LP/MILP-решатель может не найти решение из-за чрезмерного числа ограничений. Для пользовательской системы это недопустимо, поэтому реализован резервный маршрут.

### 9.2 Структура fallback

Жадный fallback работает в четыре фазы:

1. покрытие обязательных мышц;
2. выполнение минимальных требований фокус-групп;
3. заполнение оставшихся слотов с приоритетом покрытия недостающих мышц;
4. расслабление ограничений, если решение всё ещё не собрано.

Этот механизм обеспечивает деградацию качества без отказа сервиса. С инженерной точки зрения это корректный компромисс между строгостью модели и практической устойчивостью.

### 9.3 Различимость путей

В API результат помечается `usedFallback`. Это важно, поскольку клиент или аналитический слой может отличить:

1. строго оптимизированный план;
2. эвристически восстановленный план.

Таким образом, система не скрывает факт ослабления оптимальности.

## 10. Метрики как аналитический слой

Эндпоинт `POST /workout-milp/metrics` вычисляет производные показатели для уже заданных упражнений.

Основные метрики:

1. `totalSets` - суммарное число подходов;
2. `totalReps` - суммарное число повторений;
3. `estimatedTonnageKg` - оценка тоннажа;
4. `relativeIntensity` - условная интенсивность по цели;
5. `activeTimeSec` - активное время;
6. `restTimeSec` - время отдыха;
7. `estimatedCalories` - энергозатраты;
8. `muscleLoadScores` - нагрузка по мышцам;
9. `fatigueIndex` - индекс утомления.

Данный слой важен тем, что завершает цикл «оптимизация -> интерпретация». То есть MILP не только выбирает упражнения, но и предоставляет измеримые следствия решения.

## 11. Сопоставление документации и реализации

### 11.1 Консистентные элементы

Между `MILP.md`, `MILP-COMMON.md` и кодом наблюдается высокая степень согласованности по следующим направлениям:

1. `alpha_1`, `alpha_2`, `alpha_3`, `delta`, `epsilon`, `theta`, `lambda` совпадают по смыслу и значениям;
2. используется экспоненциальное восстановление усталости;
3. реализована недельная кумуляция объёма;
4. присутствуют goal-aware и gender-aware модификаторы;
5. варьируется число подходов в зависимости от типа упражнения;
6. предусмотрен fallback при неразрешимости.

### 11.2 Практические расхождения в модели

Часть формул в MILP-документации является идеализированной и описывает строгий глобальный MILP, тогда как код в ряде мест использует гибридную схему:

1. часть ограничений реализована как предварительная фильтрация кандидатов;
2. часть логики сведена к весовому ранжированию;
3. часть оптимизационной логики дополняется эвристическим fallback.

Это не ошибка, а нормальная инженерная адаптация математической модели к прикладной среде, где требуется учитывать качество, скорость и надёжность ответа.

## 12. Научная интерпретация закономерностей

### 12.1 Модель как иерархия приоритетов

Система строит решение по принципу иерархии:

1. жёсткие ограничения - безопасность, оборудование, время, противопоказания;
2. структурные ограничения - обязательные мышцы, баланс push/pull, вариативность;
3. мягкие предпочтения - цель, пол, фаза, история, разнообразие;
4. восстановительная динамика - decay усталости между сессиями.

Такое построение делает систему объяснимой: каждое решение можно отнести либо к ограничению, либо к весовому приоритету.

### 12.2 Роль мультипликативной агрегации

Большинство весов формируется через произведение факторов. Это математически означает, что качество кандидата определяется не одним признаком, а согласованностью нескольких признаков одновременно. Следовательно, упражнение будет высоко оценено только тогда, когда оно:

1. соответствует фазе;
2. не дублирует недавнюю историю;
3. не перегружает уже утомлённые мышцы;
4. соответствует цели и типу сессии;
5. не конфликтует с противопоказаниями и доступным оборудованием.

### 12.3 Рекуррентность недельной модели

Weekly Process MILP представляет собой не статическую задачу, а рекуррентную систему, где состояние следующей сессии зависит от результата предыдущей. Это сближает решение с моделью дискретного управления с памятью.

## 13. Итог

Рассматриваемое решение представляет собой зрелую гибридную архитектуру, в которой:

1. REST API обеспечивает безопасный доступ и разделение ответственности;
2. диалоговый слой переводит пользовательское намерение в формализованный вход;
3. MILP-ядро реализует оптимизацию на уровне одной тренировки и недельного плана;
4. эвристические механизмы обеспечивают устойчивость при сложных или конфликтующих ограничениях;
5. аналитический слой метрик позволяет оценить качество результата количественно.

Главный методологический вывод состоит в том, что система использует не одну глобальную формулу, а композицию нескольких моделей: логическая фильтрация, весовое ранжирование, дискретная оптимизация и рекурсивное обновление состояния. Именно такая композиция обеспечивает прикладную пригодность решения в реальной тренировочной предметной области.
